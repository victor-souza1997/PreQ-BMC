# Semantic Fidelity Audit

## Executive Answer

PreQ-BMC must verify the same fixed-point semantics it deploys. Before the semantic-fidelity fix on `feature/cbc-backend`, the deployed generated C backend and Python interpreter used saturated, rounded `__int128` fixed-point arithmetic, while the main ESBMC output harnesses used `long long` accumulation, truncating C division, and unclamped logits. That mismatch was a **BLOCKER** for an unconditional deployment-aware C-level verification claim.

After the fix, the deployed generated C backend and ESBMC output harnesses share the same generated fixed-point C kernel:

- `__int128` accumulator arithmetic;
- `div_round_half_away_from_zero_i128`;
- `clamp_to_signed_range_i128`;
- layer step order: rescale, add bias, clamp, optional hidden ReLU, final clamp.

The hidden preimage contract harness remains interval-based for scalability, but it now over-approximates the deployed rounded-and-clamped affine semantics using monotone round-half-away endpoints and deployed clamp semantics. Formal no-saturation remains available as an optional stronger property; the main contract no longer relies on no-saturation to model deployment clamp behavior.

## Findings

| Severity | Finding | Status |
| --- | --- | --- |
| BLOCKER | Output ESBMC harness classified unclamped logits computed with `long long acc` and `acc / SCALE_FACTOR`, while deployment classified rounded-and-clamped outputs. | Fixed by shared kernel in output harnesses. |
| BLOCKER | Hidden contract harness omitted deployment clamp semantics. Clamp omission can be unsound when the unclamped affine value lies inside the preimage but the clamped value does not. | Fixed by applying deployed clamp semantics to the interval image. |
| HIGH | ESBMC `paper-fast` does not enable `--overflow-check`. This is unsafe if harness arithmetic depends on overflowing `long long`. | Acceptable after fix because generated harness arithmetic casts products and accumulators to `__int128`. |
| MEDIUM | No-saturation harness uses interval floor/ceil bounds instead of exact pointwise round-half-away endpoints. | Conservative stronger property; documented as an over-approximation. |
| LOW | TensorFlow oneDNN/CUDA logs can be noisy during runs. | Not a semantic issue. |

## Deployed Fixed-Point Semantics

The deployed C backend generated by `tool/backends/c_qnn_generator.py` and the Python interpreter in `tool/backends/fixed_point.py` implement:

1. Quantized integer inputs, weights, and biases.
2. Per-neuron accumulator:
   `acc = sum((__int128)weight_int * (__int128)input_int)`.
3. Rescale by the input fractional scale using round-half-away-from-zero.
4. Add the quantized bias.
5. Clamp to the signed `Q`-bit integer range.
6. Apply ReLU for hidden layers only.
7. Clamp again to the same signed `Q`-bit range.
8. Store the final integer activation/logit in `int64_t`.

The shared C source for this arithmetic lives in `tool/verification/arith_kernel.py` and is used verbatim by both deployment generation and ESBMC templates.

## ESBMC Harness Semantics

Output classification harnesses now use the exact deployed pointwise layer step before comparing classes. This means target-class and valid-set output properties are checked over the deployed rounded-and-clamped logits.

Hidden affine contract harnesses check a conservative interval image for the preactivation affine contract. For each output neuron, they:

1. Bound the accumulator with `__int128` interval arithmetic over the input box.
2. Apply the monotone round-half-away endpoint transform.
3. Add bias.
4. Apply the deployed clamp/ReLU/clamp interval transformer.
5. Assert the resulting interval is contained in the stored preimage contract, with the existing tolerance.

This is conservative: if the interval harness verifies, every deployed rounded-and-clamped affine output for that input box is inside the preimage interval. Because it is a box interval over-approximation, it may reject some candidates that a more expensive pointwise or relational proof could verify.

No-saturation harnesses check the stronger pre-clamp property that the affine layer image stays within the representable signed range. They remain optional evidence unless the pipeline configuration requires formal no-saturation for acceptance.

The ESBMC search also raises each layer's sign-inclusive integer-width floor when the DeepPoly/preimage contract range cannot be represented by the original Quadapter weight-derived integer width. This preserves one shared `<Q,I,F>` per layer; it only prevents the search from repeatedly increasing `F` while leaving the real representable range too small for the contract.

## Required Audit Answers

1. The deployed fixed-point semantics are the generated C/Python semantics described above: `__int128` accumulation, round-half-away-from-zero rescale, bias addition, signed-range clamp, hidden ReLU, and final clamp.
2. Output ESBMC harnesses verify exact deployed output-layer arithmetic before classification; hidden ESBMC harnesses verify conservative interval containment for rounded-and-clamped affine contracts; no-saturation harnesses verify a stronger pre-clamp representability property.
3. Output harnesses are semantically identical to deployment for the checked layer step. Hidden harnesses are conservative over-approximations of the deployed affine image for the active pre-ReLU preimage obligation.
4. Before the fix, output harnesses diverged in accumulator width, rounding, clamping, and classification over unclamped logits. Hidden harnesses diverged by omitting clamp semantics.
5. The hidden interval arithmetic and no-saturation interval arithmetic are conservative. They are harmless for soundness but can produce false negatives when the box abstraction is wider than the reachable layer image.
6. The stale output harness was unsound for the paper claim because it could verify a classifier different from the deployed fixed-point classifier. Clamp omission in hidden contracts was also unsound unless formal no-saturation became a mandatory acceptance condition.
7. `paper-fast` is safe for the corrected paper claim because the main harnesses no longer rely on overflowing `long long` arithmetic. `safety` and `overflow` profiles are useful diagnostic strengthenings.
8. Before the fix, the theorem had to be conditional on an external proof that harness arithmetic matched deployment arithmetic, such as no-saturation plus rounding-equivalence assumptions.
9. After the fix, verified output properties and hidden contracts apply to the generated fixed-point C backend semantics, with hidden contracts proved through conservative interval inclusion.
10. All experiments reporting ESBMC deployment-verification statuses must be rerun because the generated harness semantics changed.

## Claim Before And After The Fix

Before this fix, the strongest honest theorem was conditional:

> If the ESBMC harness arithmetic is equivalent to the deployed arithmetic for the checked layer, for example through separately verified no-saturation plus rounding slack assumptions, then the reported contract result applies to deployment.

After this fix, the intended claim is:

> For the generated fixed-point backend semantics, ESBMC output harnesses verify the exact deployed output-layer arithmetic, and hidden ESBMC contract harnesses verify a conservative interval over-approximation of the deployed rounded-and-clamped affine arithmetic for the stored preimage contracts. Therefore verified layer contracts apply to the generated deployment C backend under the selected shared layer `<Q,I,F>`.

## ESBMC Profile Policy

`paper-fast` is safe for the paper claim after this fix because the generated harness arithmetic no longer relies on overflowing `long long` multiplication or accumulation. Products and accumulators are promoted to `__int128`, and division is guarded by assertions that the scale denominator is positive.

Use `overflow` or `safety` profiles for diagnostic strengthening when desired. They are not required to compensate for stale `long long` overflow in the main harnesses after this fix.

Unsafe profile/configuration combinations:

- Any external or legacy harness that still performs fixed-point products in `long long` without `--overflow-check`.
- Any manually edited harness that removes the shared kernel clamp or round-half-away helpers while keeping deployment-aware claims.

## CBC Soundness Note

CBC is a candidate-generation backend for MILP preimage synthesis and MILP forward checks. Final soundness for deployment-aware claims should rest on ESBMC verification of the generated fixed-point C contracts, not on a claim that CBC is equivalent to Gurobi.

CBC and Gurobi may differ in performance, numerical tolerances, and branch-and-bound choices. Large article results should record which solver backend generated candidate preimages and bit-widths. CBC improves artifact reproducibility because it is license-free, but it should not be claimed equivalent to Gurobi without empirical parity tests for the relevant benchmark set.

## Compositional Soundness Claim

Informally, a fixed-point layer semantics maps an integer input box through the generated C layer step: `__int128` accumulation, round-half-away rescale, bias, clamp, optional ReLU, and final clamp. A layer input contract is the integer region assumed at the layer input. A layer output/preimage contract is the integer region that the next layer or output property requires. The ESBMC obligation is to prove that every deployed layer output produced from the input contract lies inside the output/preimage contract.

Composition lemma:

> If every layer contract is verified under the same fixed-point semantics as the deployed C backend, and the output property harness verifies the final classification condition, then the deployed fixed-point network satisfies the local property for all inputs in the initial region.

Limitations:

- The claim is local to the selected input region, not global robustness.
- The claim is relative to the generated C semantics and the ESBMC/Bitwuzla result.
- `UNKNOWN`, `TIMEOUT`, and `MEMOUT` are not proofs.
- Empirical diagnostics are not formal guarantees.
